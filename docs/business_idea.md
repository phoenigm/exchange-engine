### 1. Модель данных (как хранятся заявки)

В памяти биржи для каждой торговой пары (например, `BTC/USDT`) существуют две отсортированные структуры данных:

*   **Bids (на покупку):** `Max-Heap` или `SortedList` по убыванию цены. Самый большой спрос («лучший бид») — первый.
*   **Asks (на продажу):** `Min-Heap` или `SortedList` по возрастанию цены. Самая низкая цена предложения («лучший аск») — первый.

Каждый ордер — это объект: `{ id, price, quantity, timestamp, type, user_id }`.

### 2. Алгоритм сопоставления (Matching Engine)

«Текущая цена» токена — это не самостоятельная сущность, а **побочный эффект** исполнившегося пересечения заявок. Цикл обработки при поступлении нового ордера (например, Market Buy):

1.  **Блокировка пары:** Берется мьютекс на `order_book_BTCUSDT`.
2.  **Проверка встречных ордеров:**
    *   Пока `new_order.quantity > 0` и `top_ask.price <= new_order.price` (или для маркета всегда).
    *   Снимаем встречный ордер `top_ask`.
    *   Считаем сделку (trade):
        *   `price = top_ask.price`
        *   `quantity = min(new_order.quantity, top_ask.quantity)`
        *   `timestamp = now`
    *   Фиксируем **последнюю цену (last_price)** = `price`.
3.  **Частичное заполнение:** Уменьшаем `quantity` в одном ордере, обновляем `top_ask` или удаляем его при `quantity == 0`.
4.  Если остатки `new_order.quantity > 0` и ордер лимитный — он попадает в стакан как новый `bid`.

**Ключевой вывод:** Цена сделки — это всегда цена лимитного ордера, который ждал в стакане. Маркет-ордер только «соглашается» с ней.

### 3. Откуда пользователи видят «текущую цену» (топ 5, последняя)

Дешборд клиента получает агрегаты:

*   **Last price:** `Cache(last_trade.price)`
*   **Mark price** (индекс для фьючерсов): Взвешенная средняя нескольких spot бирж, чтобы избежать манипуляции одной биржей.
*   **Index price:** Используется для ликвидаций. Не из стакана.
*   **VWAP (Volume Weighted Average Price):** Сумма `(price * quantity)` за период / сумма `quantity`.

### 4. Что такое Ликвидность и проскальзывание (Slippage)

Реальная цена покупки `N` токенов вычисляется интегралом по кривой ликвидности:

`Cost(N) = Sum(ask_i.price * min(ask_i.quantity, remaining_N))`

**Пример для программиста:**
Стакан `asks`: `[(100, 1), (101, 1), (102, 10)]`

Покупка 1.5 токена:
*   Покупаем 1 токен по 100
*   Покупаем 0.5 токена по 101
*   **Средняя цена** = (1*100 + 0.5*101) / 1.5 = **100.333**

Если писать драйвер для торговли, ты ЭТО и будешь рассчитывать — никогда не верь, что сможешь купить по `top_ask.price` полный объем.

### 5. Факторы, искажающие «естественную» цену

Программист должен знать про механизмы, которые ломают простую модель:

1.  **Фикс спреда (Spread):** Биржа не может исполнить сделку, пока `top_ask.price > top_bid.price`. Пользователи теряют разницу (например, в `BTC/USDT`: спред 0.5 USDT).
2.  **Фантомные ордера (Iceberg):** Видишь только вершину. Внутренняя логика при частичном исполнении докидывает скрытый объем.
3.  **Комиссии (Maker / Taker):**
    *   *Maker* (поставил лимитник в стакан) — платит меньше или получает кэшбэк (стимуляция ликвидности).
    *   *Taker* (снял маркетом или пересек стакан) — платит больше.
    *   Программно цена исполнения `price_exec` всегда превращается в `price_exec ± fee`.
4.  **Книга отмен (Cancel book):** Некоторые стаканы поддерживают отложенные ордера (Timed order). По таймеру `price` блокируется, а ордер удаляется до начала обработки нового блока транзакций. Это защита от front-running на уровне движка.

### Сравнение с DEX (для понимания границ)

Если ты спросишь про Uniswap (AMM — Automated Market Maker), там работает формула:
`x * y = k`
`price_token0 = reserve_token0 / reserve_token1`
И цена огромного ордера вычисляется как:
`output_token1 = reserve_token1 - (k / (reserve_token0 + input_token0))`

Там цена **не множится на объем**, а считается квадратично. Это важно, чтобы не путать CEX (лоббинг) и DEX (формула).

### Итог для программиста (как системная модель)

> Цена токена на CEX — это неустойчивое **состояние**, производное от пересечения двух приоритетных очередей (bids и asks). Алгоритмически это `(heap_min_ask + heap_max_bid) / 2` при нулевой активности, либо значение последнего успешного `pop` из `min_ask`, когда `ask.price <= taker_bid.price`.

**Где живет код:** в `MatchingEngine::processOrder()` + `OrderBook::match()` с O(log N) на вставку/удаление через сбалансированное дерево (как `std::map`).
